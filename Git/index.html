¿Qué es Git?

Git es un sistema de control de versiones distribuido: registra los cambios de tus archivos a lo largo del tiempo y te permite volver a versiones anteriores, comparar cambios y trabajar en ramas sin pisar el trabajo de otros.

No es lo mismo que GitHub: GitHub es un servicio en la nube que aloja repositorios Git. Git es la herramienta que usas en tu máquina; GitHub es un lugar donde subir y compartir esos repositorios.

Configuración inicial

Identidad y editor
git config --global user.name "Carlos Galindo"
Define el nombre que aparecerá en todos tus commits. Es lo primero que debes configurar; Git lo usa para saber quién hizo cada cambio.
git config --global user.email "carlosxgalindo@gmail.com"
Define el email asociado a tus commits. Debe ser el mismo que uses en GitHub u otros servicios si vas a subir el repositorio.
git config --global -e
Abre el archivo de configuración global para ver/editar valores.
Alias útiles
git config --global alias.s "status --short"
Atajo: git s para ver estado resumido.
git config --global alias.lg "log --graph --abbrev-commit --decorate --format=..."
Atajo: git lg para un log gráfico y legible.

Iniciar repositorio y primer commit

git init
Convierte la carpeta actual en un repositorio Git creando la carpeta oculta .git. Es el primer comando que usas en un proyecto nuevo; solo se ejecuta una vez por proyecto.
git config --global init.defaultBranch main
Hace que la rama por defecto sea main en lugar de master.
git add .
Prepara todos los archivos modificados o nuevos para el próximo commit (los pone en “Stage”). Sin este paso, git commit no incluiría los cambios. El punto significa “todo el directorio actual”.
git commit -m "Primer Commit"
Guarda en el historial una “foto” de todo lo que está en Stage, con el mensaje que pongas entre comillas. Es la forma de ir dejando versiones guardadas de tu proyecto.

Ramas básicas y archivos en el tiempo

git checkout -b branch-nombre
Crea una nueva rama con ese nombre y te cambia a ella de inmediato. Muy usado para empezar una funcionalidad o corrección sin tocar main.
git checkout <hash> archivo.ext
Recupera ese archivo tal como estaba en el commit con ese hash (útil para “abrir” el archivo en ese momento).

Comandos básicos (1)

git checkout -- .
Muy importante: descarta todos los cambios no guardados en el directorio actual y deja los archivos como en el último commit. Útil cuando quieres “tirar” todo lo que modificaste y volver al estado limpio; ten cuidado porque no se puede deshacer.
git branch
Lista todas las ramas del repositorio; el asterisco (*) indica en cuál estás trabajando. Útil para no perderte cuando tienes varias ramas.
git commit -am "Updating README"
Stage y commit en un comando (solo para archivos ya rastreados).
git log
Muestra el historial completo de commits: autor, fecha y mensaje. Es la forma de ver qué se ha guardado y en qué orden. Con q sales de la vista.

Comandos básicos (2)

git add folder/.gitkeep
Añade un .gitkeep dentro de una carpeta vacía para que Git la rastree (Git no versiona carpetas vacías).
git diff
Muestra línea por línea qué has modificado en los archivos que aún no has puesto en Stage. Sirve para revisar antes de hacer git add y commit.
git commit --amend -m "Mensaje modificado"
Actualiza el mensaje del último commit.
git reset --soft HEAD^
Deshace el último commit pero deja los cambios en Stage (HEAD^ = anterior; HEAD^2, HEAD^3 para más atrás).

Comandos básicos (3)

git reset --mixed bc3721a
Elimina commits hasta ese hash; los cambios quedan en el árbol de trabajo, no en Stage.
git reset --hard 22cc4ee
Deja el repositorio exactamente como en ese commit (pierdes cambios posteriores).
git reflog
Muestra el historial de referencias; para “viajar” a un punto: git reset --hard <hash>.
git mv destroy.md save.md / git rm save.md
Renombrar y eliminar archivos de forma rastreada por Git.
.gitignore
Archivo de texto en la raíz del proyecto donde pones nombres de archivos o carpetas que Git debe ignorar (por ejemplo node_modules/, .env, archivos temporales). Evita subir cosas que no deben versionarse.

Ramas: crear, cambiar y fusionar

git branch branch-nombre
Crea una nueva rama (no cambia a ella).
git checkout branch-nombre / git checkout main
Te mueve a la rama indicada. Todo lo que hagas a partir de entonces (commits, cambios) quedará en esa rama hasta que vuelvas a cambiar.

Importante: hay que estar en la rama donde quieres incorporar los cambios (por ejemplo main).

git merge branch-nombre
Integra en la rama actual (por ejemplo main) todos los commits de la otra rama. Primero te pones en main con checkout main, luego ejecutas merge con el nombre de la rama a traer.
git branch -d branch-nombre
Elimina una rama ya fusionada (desocupada).

Tags (releases)

Los tags marcan puntos concretos (por ejemplo releases). Dos ejemplos:

git tag v1.0.0 · git tag
Crear un tag simple en el commit actual y listar todos los tags.
git tag -a v1.0.0 -m "ver 1.0.0 ready"
Tag anotado con mensaje (recomendado para releases).
git tag -a v0.1.0 <hash> -m "ver alpha"
Tag anotado en un commit pasado; git show v1.0.0 muestra la info del tag.

Stash (guardado temporal)

Stash es un espacio temporal para guardar trabajo en progreso (WIP) sin hacer commit.

git stash
Guarda los cambios actuales en un stash.
git stash list
Muestra la lista de stashes (comando correcto: git stash list, no gist).
git stash pop
Aplica y elimina el último stash.
git stash clear
Borra todos los stashes.

Rebase

Rebase sirve para recolocar los commits de una rama sobre otra (por ejemplo, “mover” tus commits encima de los de main), reescribiendo la línea del tiempo.

git rebase HEAD~2
Reaplica los últimos 2 commits (incorpora cambios de main en tu rama, según contexto).
git rebase -i HEAD~4
Modo interactivo: editar, reordenar o unir (squash) los últimos 4 commits. s <hash> une commits.
git rebase -i HEAD~3 → r <hash>
Para reword (actualizar mensajes de commit).
git rebase --continue
Continúa el rebase después de resolver conflictos o ediciones.

El número en HEAD~X indica cuántos commits participan en el rebase.

Árbol y ramas en tu computadora

Repositorio local (.git): ramas, commits y merge.

C1→C2: main · C3→C4: feature · merge une feature en main.
